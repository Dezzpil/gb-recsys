# Рекомендательная система для GamersBase (gb-recsys)

Проект по разработке и оценке эффективности рекомендательной системы для игрового маркетплейса GamersBase. Основная цель проекта — повышение вовлеченности пользователей и персонализация опыта покупки игр на основе исторических данных.

### Цели проекта
- Разработка масштабируемой и эффективной системы рекомендаций для интернет-магазина видеоигр.
- Повышение ключевых метрик вовлеченности (CTR, конверсия в покупку) за счет персонализации контента.
- Исследование влияния различных типов данных (транзакционные и поведенческие) на качество рекомендаций.

### Основные задачи
1. **Сбор и подготовка данных**:
    - Выгрузка транзакционных данных о покупках пользователей.
    - Автоматизация сбора сырых логов поведения пользователей (просмотры, показы товаров) через API Яндекс Метрики (`04-metrika-pipeline.ipynb`).
    - Очистка и предобработка данных, приведение их к формату, совместимому с современными библиотеками (например, `rectools`).
2. **Исследование алгоритмов рекомендаций**:
    - Реализация базовых алгоритмов коллаборативной фильтрации (User-based CF) для быстрой проверки гипотез (`01-cf.ipynb`).
    - Сравнение классических и современных моделей машинного обучения (ALS, BPR, LightFM, PureSVD, EASE) с использованием библиотеки `rectools` (`05-rectools-baselines.ipynb`).
3. **Оценка качества (Evaluation Pipeline)**:
    - Настройка системы кросс-валидации с учетом временной составляющей (Time-based split) для имитации реальной работы системы.
    - Использование комплексного набора метрик:
        - **Метрики точности**: Precision@k, Recall@k, NDCG@k.
        - **Метрики разнообразия и новизны**: MeanInvUserFreq (Novelty), Serendipity.

### Используемый подход и стек технологий
- **Данные**:
    - Транзакционные данные (CSV, SQL): информация о купленных ключах и товарах.
    - Данные Яндекс Метрики (Logs API): детальные логи взаимодействия пользователей с сайтом.
- **Библиотеки**:
    - `pandas`, `numpy`, `sklearn` — базовая обработка данных.
    - `implicit`, `lightfm`, `rectools` — построение и оценка моделей рекомендательных систем.
    - `requests` — взаимодействие с внешними API.

### Использование скрипта обогащения данных
Для добавления случайных продуктов в файл покупок (1-3 продукта на заказ с распределением 60%/30%/10%):
```bash
python3 add_products_to_orders.py assets/20260302-gb-orders.csv assets/20251012-gb-products.csv
```
Результат будет сохранен в `data/orders` с суффиксом `-with-products.csv`.

### Использование скрипта выгрузки Яндекс.Метрики
Скрипт `app/export_metrika.py` позволяет выгружать логи визитов из Яндекс.Метрики за указанный период. По умолчанию выгружаются данные за последние 3 года.
Пример запуска:
```bash
# Выгрузка за 3 года (по умолчанию)
PYTHONPATH=. python3 app/export_metrika.py

# Выгрузка за конкретный период
PYTHONPATH=. python3 app/export_metrika.py --start 2024-01-01 --end 2024-12-31
```
Результаты сохраняются в директорию, указанную в `METRIKA_DATA_DIR` (по умолчанию `data/metrika`), а информация о выгрузке логируется в таблицу `metrika_logs` в БД.

### Объединение данных заказов и Яндекс.Метрики
Скрипт `app/merge_metrika_orders.py` связывает пользователей с историей их посещения сайта и покупок.
Логика объединения:
1. Загружаются последние файлы из `data/metrika`, `data/orders` и `data/products`.
2. Данные Метрики и заказов матчатся по `Last Ymcid` и `clientID`.
3. Для каждого email собирается список просмотренных продуктов (из Метрики) и купленных (из заказов).
4. Продукты приводятся к `id` из справочника продуктов:
    - Покупки матчатся по `skuId`.
    - Просмотры матчатся по `product_name`.
5. Каждому продукту присваивается вес:
    - Просмотр (из Метрики) = 0.5.
    - Покупка (из заказов) = 1.0.
    - Если продукт был и просмотрен, и куплен, выбирается вес 1.0.
6. Продукты, не найденные в справочнике, игнорируются и сохраняются в `data/merged/DATE_missing_products.txt`.

Пример запуска:
```bash
PYTHONPATH=. python3 app/merge_metrika_orders.py
```
Результат сохраняется в `data/merged/DATE_email_products.csv` в формате `email, product_id, weight`. Информация о процессе логируется в таблицу `merge_logs` в БД.

### Обучение моделей рекомендаций
Для обучения моделей используются классы в директории `app/models/`. Каждая модель наследуется от `BaseModel` и реализует методы `fit()` и `predict()`. Результаты обучения сохраняются в таблицу `recommendations` в БД и привязываются к `merge_log_id`.

#### 1. User-based Collaborative Filtering (CF)
- **Файл**: `app/models/cf.py`.
- **Алгоритм**: Использует данные из таблицы `user_interactions` для расчета сходства пользователей по косинусному расстоянию. Рекомендует товары, которые покупали похожие пользователи, но которых нет у текущего.
- **Запуск обучения**:
  ```bash
  PYTHONPATH=. python3 app/train_cf.py
  ```

#### 2. Similar Games (Content-based)
- **Файл**: `app/models/similar.py`.
- **Алгоритм**: Основан на поле `similar` в справочнике товаров (`*products*.csv`). Для каждого купленного или просмотренного пользователем товара находятся связанные игры. Вес рекомендации зависит от частоты упоминания игры в списке похожих для всех взаимодействий пользователя.
- **Запуск обучения**:
  ```bash
  PYTHONPATH=. python3 app/train_similar.py
  ```

#### 3. Steam API Recommender
- **Файл**: `app/models/steam.py`.
- **Алгоритм**: Использует внешний Steam API для поиска похожих игр. 
    1. Модель отправляет названия игр, с которыми взаимодействовал пользователь (покупки и просмотры), во внешнюю систему батчами.
    2. Для получения асинхронных ответов внутри модели запускается временный FastAPI-сервер (callback).
    3. Полученные рекомендации фильтруются по наличию в локальном справочнике товаров.
    4. Вес рассчитывается на основе частоты появления игры в ответах и её позиции в выдаче API.
- **Требования**: Необходимо наличие запущенного внешнего API (адрес настраивается в `.env`).
- **Запуск обучения**:
  ```bash
  PYTHONPATH=. python3 app/train_steam.py
  ```

### Полный цикл обработки (Pipeline)

Для автоматизации всего процесса от выгрузки данных до генерации рекомендаций используется скрипт `app/pipeline.py`. 

Pipeline последовательно выполняет следующие шаги:
1. **Выгрузка из Яндекс.Метрики**: Получение свежих логов поведения пользователей.
2. **Объединение данных**: Матчинг Метрики с заказами, расчет весов взаимодействий и сохранение в `user_interactions`.
3. **Генерация графиков**: Создание визуализации распределения популярности товаров для текущего объединения.
4. **Обучение моделей**: Последовательный запуск обучения всех доступных моделей (`user_based_cf`, `similar_games`, `steam`).
5. **Логирование**: Сохранение детальной статистики обучения (время старта, длительность, количество рекомендаций, статус) в таблицу `model_training_logs`.

Пример запуска:
```bash
# Запуск полного цикла
python3 app/pipeline.py

# Запуск без выгрузки из Метрики (используя существующие файлы)
python3 app/pipeline.py --skip-metrika

# Ограничение периода выгрузки Метрики
python3 app/pipeline.py --start 2024-01-01 --end 2024-12-31
```

### Настройка базы данных и миграции

Для работы с базой данных используется PostgreSQL v17, запускаемая через Docker Compose, и Alembic для управления миграциями.

#### Подготовка окружения
Скопируйте `.env.example` в `.env` и при необходимости измените реквизиты доступа:
```bash
cp .env.example .env
```

#### Запуск базы данных
Для запуска PostgreSQL в контейнере выполните:
```bash
docker-compose up -d
```

#### Управление миграциями
Alembic используется для обновления схемы базы данных.

- **Применить все миграции до актуальной версии**:
  ```bash
  alembic upgrade head
  ```

- **Создать новую миграцию (после изменения моделей)**:
  ```bash
  alembic revision --autogenerate -m "description of changes"
  ```

- **Откатить последнюю миграцию**:
  ```bash
  alembic downgrade -1
  ```

- **Посмотреть текущий статус миграций**:
  ```bash
  alembic current
  ```

### Запуск интерфейсов

Для мониторинга процессов и просмотра рекомендаций доступны API и веб-интерфейс.

#### 1. Запуск API
API предоставляет эндпоинты для получения статистики объединений, списка пользователей и их персональных рекомендаций.
```bash
PYTHONPATH=. python3 app/api.py
```
API будет доступно по адресу `http://localhost:4000`. Документация (Swagger) доступна по адресу `http://localhost:4000/docs`.

#### 2. Запуск веб-интерфейса
Веб-интерфейс позволяет визуально анализировать результаты работы системы.
```bash
cd web
npm install  # если запускается впервые
npm run dev
```
Интерфейс будет доступен по адресу `http://localhost:5173` (или другому порту, указанному в консоли Vite). Он автоматически проксирует запросы к API на порту 4000.
